## Overview

The public roadmap shows what content is planned, what is actively in development, and what has shipped. Tracking it tells you what to expect next — and gives you the right place to influence priorities by upvoting items or opening new Issues. Everything from new industry use cases to additional update sets and certification prep resources flows through the same GitHub-based roadmap process.

## Key concepts

* **GitHub Projects (Roadmap board)** — The roadmap is maintained as a GitHub Projects board in the companion repository. It has four columns: Backlog (identified but not scheduled), In Progress (actively being developed), In Review (in PR review or beta), and Shipped (released). You can view it without a GitHub account; to comment or upvote, a free GitHub account is required.
* **Issues as roadmap items** — Each roadmap card links to a GitHub Issue. The Issue contains the detailed scope, any design decisions, and the discussion thread. Following the Issue gives you notifications when status changes.
* **Milestones** — Issues are grouped into Milestones that correspond to handbook releases (e.g., `v1.0 — IRM Foundations`, `v1.1 — Advanced Risk`, `v1.2 — Industry Use Cases`). Milestones give a rough release timeline and make it easy to see what ships together.
* **Issue types** — Issues are labeled to indicate type: `bug` (error in existing content or an update set), `content-gap` (missing coverage of a topic), `enhancement` (improvement to existing content), `update-set` (new or revised update set needed), `diagram` (new or revised architecture diagram), `cert-prep` (certification study resource gap).
* **Priority signals** — Issue priority is influenced by: GitHub reaction count (thumbs up on the Issue), comments with specific use-case context, and alignment with the overall handbook structure. Niche requests with broad applicability are prioritized over highly specific one-off asks.

## How to Raise an Issue

1. Check the existing Issues first — search by keyword before opening a new one.
2. Click **New Issue** in the GitHub repository and choose the appropriate template.
3. Fill in the title (concise, specific) and description (problem or gap, why it matters, any suggested approach).
4. Add relevant labels if you have write access; otherwise the maintainer will label it on triage.
5. Reference any related Issues or PRs in the description.

{% hint style="info" %}
The most actionable Issues are those that describe a specific gap with a concrete use case: "The CIS-RC cert prep section doesn't cover the Audit Management exam domain, which represents approximately 15% of exam questions." That is easy to prioritize. Vague requests ("add more content") are deprioritized.
{% endhint %}

## Key takeaways

* The roadmap is public — check it before asking what's coming next.
* Upvoting and commenting on existing Issues is the fastest way to influence prioritization.
* Every roadmap card links to a GitHub Issue with the full discussion — follow the Issue, not just the board card.
* Specific, use-case-grounded Issues move faster than general requests.
* Milestones show the release grouping — items in a milestone ship together.
